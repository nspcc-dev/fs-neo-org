Features

NeoFS is a distributed, decentralized object storage network developed by Neo SPCC. Built with Neo blockchain integration in mind, NeoFS aims to support the shift away from third-party storage providers, providing users with complete control over their data.

Many of nodes, organized in peer-to-peer network that take care of storing and distributing user's data. Any Neo user may participate in NeoFS network and get payed for providing storage resources to other users, or store his data in NeoFS and pay competitive price for it.

The service is designed to work with NEO smart contracts, allowing for truly decentralized applications, and can also be used as a content delivery network. Users can rent out storage in return for Neo GAS tokens, or use GAS to store files on the network.

Representing NEO’s commitment to building the future, NeoFS offers unparalleled data storage control to users and developers by bringing together the benefits of decentralization and real-world integration.

NeoFS ensures user data security and mitigates against malignant nodes by ensuring zero-knowledge data validation through a combination of homomorphic cryptography and game theory models.

NeoFS puts the control over data in users’ hands

Because anyone may participate in NeoFS network, we need a reliable mechanisms to ensure users' data consistency and availability in a very chaotic environment.

To achieve this goal we use a combination of homomorphic hashing for data integrity checks and game theory models to create a situation where all nodes behave correctly and the data is stored safely.

The decentralized nature of NeoFS eliminates not only the centralized points of failure, but also centralized points of control. In NeoFS we try to shift as much power to user's side, as possible. only users have full control over their data, where, how it's stored, and who can get access to it.

NeoFS is geared for real-world applications

Unlike many other projects in this area, we are not trying to create a parallel reality, in a technological sense. We understand that for solving real-world problems we need to use standard protocols and have ability to integrate NeoFS with existing systems. There are protocol adapters, or gateways, that allow access NeoFS via standard HTTP or S3 protocols. Users can develop their own gateways to integrate NeoFS with existing systems they are using, like accounting system or CRM or maybe a corporate backup service.

Almost infinitely scalable due to NetworkMap and Data Placement

We have almost infinitely horizontal scalability by our architecture features like unique Data Audit algorithm, Data Placement (possibility to get objects without request for any "master" nodes) and Network Map.

Network Map is a multi-dimensional graph where nodes have attributes and are grouped by those attributes and their values. This allows us to use a special data placement function to find nodes that would store an object when we want to put it into NeoFS network or to get it.

This approach allows us not to have any centralized meta-data storage having object's locations and not to re-balance data with every joining or leaving storage node, like it happens in DHT-based systems.

Designed to work reliably in chaotic environment

That the bigger the NeoFS network grows, the more stable it becomes, because the chance of network changed would affect the particular container are decreasing with the increasing number of nodes in the network. It means, unlike in DHT approach, the amount of data migration we need to do decreases with network growth.

Zero-knowledge data validation based on homomorphic cryptography

When you have a large number of objects in a distributed network of untrusted nodes with an ever-changing topology, you can't use classical approach with comparing objects' hashes with some sample in a central meta-data storage. This would cause unacceptable overhead.

To solve this problem, in NeoFS we use Homomorphic hasing. It's special kind of hashing algorithms that allow to compute the hash of a composite block from the hashes of the individual blocks.

Directly accessible through Smart-Contract code

In the beginning we mentioned that NeoFS can be accessed directly from SmartContract code. This feature becomes available in Neo 3.0 release with introduction of Oracles Protocol. SmartContracts will be able to issue GET and PUT requests for fixed-size objects.

GAS is used for payments rather than custom tokens

NeoFS uses GAS for all payments. To be able to pay for storage services in NeoFS, the one has to deposit some GAS to NeoFS SmartContract address. This event will be monitored by InnerRing nodes and reflected in user's account balance internally in NeoFS.

NeoFS InnerRing nodes keep track of all account using NeoFS and using dBFT2.0 consensus algorithm, the same as in Neo blockchain, agree on users' mutual settlements, reward distribution between node owners and other payment-related procedures.

Incentive model based on market principles

For incentive model and pricing we implement a free market. Each node declares how much it wants to receive as a reward for data storage services for the upcoming epoch. Placement function considers the declared price and prefers nodes with better prices. However, because nodes in NetworkMap declare key-value attribute describing their parameters, like geographical location, type of storage, capacity and other things also used in StoragePolicy, it leaves room for nodes that want higher price better or unique services to still get into the placement function result.

In short, if you are too greedy, you will not get users and will not get rewards. To have higher prices you need to satisfy market demand or provide something unique.

As you see in the example graph of price to capacity change, node needs to keep price "in market" to get most profit. this should guarantee the competitive price for data storage in NeoFS.
